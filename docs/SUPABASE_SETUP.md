# Supabase Setup

PotterPulse can run in two modes:

- **Local mode:** SQLite reads and writes through `potter_pulse.db`.
- **Optional cloud mode:** Supabase PostgREST reads squad, fixtures, and transfers when `SUPABASE_URL` and `SUPABASE_ANON_KEY` are present.

Votes, forum threads, comments, match stats, and player ratings still use the local SQLite endpoints unless those handlers are extended later.

## 1. Create Tables

Create a Supabase project, open the SQL Editor, and run:

```sql
create table if not exists stoke_squad (
  id bigint generated by default as identity primary key,
  player_name text not null,
  position text not null,
  squad_number integer not null unique
);

create table if not exists efl_fixtures (
  id bigint generated by default as identity primary key,
  opponent text not null,
  match_date date not null,
  competition text not null,
  venue text not null,
  status text default 'scheduled',
  stoke_score integer,
  opponent_score integer,
  unique (opponent, match_date)
);

create table if not exists stoke_transfers (
  id bigint generated by default as identity primary key,
  player_name text not null,
  direction text not null,
  details text not null,
  unique (player_name, direction, details)
);
```

## 2. Add Local Credentials

Copy `.env.example` to `.env`, then fill in:

```text
SUPABASE_URL=https://your-project-id.supabase.co
SUPABASE_ANON_KEY=your-supabase-anon-key
SUPABASE_SERVICE_ROLE_KEY=optional-local-migration-key
```

Do not commit `.env`. It is ignored by Git. Keep service-role keys local or in trusted deployment secrets only; never put them in frontend code.

## 3. Migrate Local Data

Run:

```powershell
node scripts/migrate-to-supabase.mjs
```

The script reads `potter_pulse.db` and posts the current squad, fixtures, and transfers into Supabase. Use a service-role key for one-off migration if row-level security blocks inserts with the anon key.

## 4. Run The App

```powershell
$env:PORT="4179"; node scripts/server.mjs
```

With Supabase keys present, the rendered page reads squad, fixtures, and transfers from Supabase. Without keys, it falls back to SQLite.

## 5. Deployment Notes

For hosted deployment, set these as platform secrets/environment variables:

- `SUPABASE_URL`
- `SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY` only for trusted one-off migration jobs, never browser code
- `FOOTBALL_API_KEY` if live fixture syncing is enabled

Never expose service-role keys in the browser or README.
